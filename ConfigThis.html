ConfigMake and ConfigThis

One-line Explanation

Manage and execute Linux testing and reporting programs through these bash shell scripts employing a directory structure containing a symbolic link to each executable file.

Two-line Explanation

These bash shell scripts are intended to iteratively load executable files via symbolic links and echo output to a log file for later review. These bash shell scripts rely on a directory structure intended to group testing and reporting links by project, target platform, and version.

Execution cycle

  1. ConfigMake
    1. ConfigMake - create vendor / category directories
    2. ConfigMake - create executable files
    3. ConfigMake - create project / platform _ version directories
    4. ConfigMake - create project links
  2. ConfigThis
    1. ConfigThis - self check - finds specified directories and related files
    2. ConfigThis - load each link
    3. ConfigThis - redirect output to a project / platform _ version log file
    4. ConfigThis - check the exit code and generate a stop file upon failure
    5. Review the log file and update target software and systems
    6. Repeat

ConfigThis will continue loading each link in each category. If a link returns a non-zero value (i.e. it terminates abnormally) then ConfigThis creates a stop file in the current project / platform _ version / category directory.

If all links complete without abnormal termination then ConfigThis creates a last.stop file in the current category directory then continues to the next category directory.

You may now review the ConfigThis log file and make adjustments to the target software and systems.

ConfigThis again. If a stop file exists then start with the indicated link in that category, or if a last stop file exists then skip the category. You may execute ConfigThis with the -nostop option causing ConfigThis to delete the stop files and then load all links per category in all categories.

Installation

  1. Download the five ConfigMake and ConfigThis files (e.g. to /home/loginID/Downloads)
    1. ConfigMake (executable script)
    2. ConfigThis (executable script)
    3. ConfigMake.txt ( ConfigMake -help )
    4. ConfigThis.txt ( ConfigThis -help )
    5. ConfigThis.html ( you are reading it now )
  2. Enable execution of ConfigMake
    • cd /home/loginID/Downloads (substitute the appropriate loginID)
    • chmod u+x ./ConfigMake

Description

  • ConfigMake is a simple and elegant Linux bash command line tool used to create and populate the project and vendor directory structures containing executable files that perform tests and generate reports.
  • ConfigThis is a simple and elegant Linux bash command line tool employing a directory-based and file-based template to load executable files via symbolic links that perform tests and generate reports.
  • ConfigThis is not designed to reinvent the wheel. It is understood that dozens of exploit frameworks, hardening frameworks, and unit testing frameworks are in use today. And, they do not talk to each other... You may leverage these tools within the ConfigThis template system
    • Download and configure scripts for use in ConfigThis
    • Create links to exploit frameworks that have APIs / CLIs (e.g. OpenVAS / OMP) to execute individual tests, one test per link
    • Create links to unit testing frameworks to execute individual unit tests, one test per link or one application per link
  • ConfigThis loads vendor-provided or user-coded stand-alone microservice-styled executable files (e.g. bash scripts, PHP, Python, Ruby, Lisp, C/C++, etc.) via links (i.e. ln), each executable file performs a test or procedure and generates corresponding echo-like statement(s). echo output is redirected to a log file, for later analysis.
  • ConfigThis directories and files are kept separate from target application, utility, and/or project files - there is one set of project directories containing links (i.e. ln) to another set of vendor directories and user-specified directories containing executable files
  • ConfigThis loads only those links found in the specified project / platform _ version directory - loaded in a sort order intrinsically determined by category directory names and link names (i.e. no metadata files).
  • ConfigThis vendor executable files may be maintained as needed without adversely impacting project tests or reports as long as vendor category directories and executable files are not deleted nor renamed. New executable files added to vendor directories will not be employed until project links are created to them.

ConfigMake Usage

  1. Prerequisites
    • Presumes installation
    • Presumes that directory /home/loginID exists (substitute the appropriate loginID)
    • Presumes that directory /home/loginID/Downloads exists
  2. Change current working directory
    • cd /home/loginID/Downloads
  3. Execute ConfigMake to create template directories
    • ./ConfigMake -this /home/loginID
  4. Change current working directory
    • cd /home/loginID
  5. Execute ConfigMake to display the contents of the help file /usr/share/ConfigMake.txt
    • ConfigMake
      • This presumes execution from /usr/local/bin
  6. Execute ConfigMake to display the contents of the conf file /etc/ConfigThis/ConfigThis.conf
    • ConfigMake -conf
  7. Execute ConfigMake to list vendor directories
    • ConfigMake -vendors
      • ./ConfigThis/vendor/aVendor/
      • ./ConfigThis/vendor/loginID/
      • ./ConfigThis/vendor/bossss/
  8. Execute ConfigMake to create vendor directories
    • ConfigMake -vendor aVendor
      • ./ConfigThis/vendor/aVendor/config_1000_os/
      • ./ConfigThis/vendor/aVendor/config_2000_ip/
      • ./ConfigThis/vendor/aVendor/config_3000_usr/
    • If the vendor directory exists then the executable files stored therein are listed
  9. Execute ConfigMake to list all vendor-supplied executable files
    • ConfigMake -files aVendor | less
    • ConfigMake -files aVendor aCategory | less
      • List all executable files for one vendor located in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/vendor/aVendor/aCategory1/aSourceFile3.sh
        • /usr/src/ConfigThis/vendor/aVendor/aCategory2/aSourceFile4.sh
      • List all executable files for one vendor in one category located in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/vendor/aVendor/aCategory/aSourceFile1.sh
        • /usr/src/ConfigThis/vendor/aVendor/aCategory/aSourceFile2.sh
      • The | less option enables paging of screen output.
  10. Execute ConfigMake to work with downloaded vendor-supplied executable files or create your own executable files
    • ConfigMake -file aSourceFile.sh aVendor aCategory
    • ConfigMake -file /usr/local/bin/aSourceFile.sh
      • Create an executable file in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/vendor/aVendor/aCategory/aSourceFile.sh
      • Else, create an executable file in the current working directory if it contains a ConfigThis directory.
        • ./ConfigThis/vendor/aVendor/aCategory/aSourceFile.sh
      • Else, create an executable file in the directory specified and the aVendor and aCategory parameters are ignored
        • /usr/local/bin/aSourceFile.sh
      • If the file exists then the system presents a prompt to edit the file using the default editor (i.e. vim), respond Y or N
      • This is where all of the real work gets done
        • A library of 500+ executable files is available on-line
        • There is nothing special about each executable file other than it must work without failure (or at least trap errors)
        • The contents of each executable file may be compiled C, C++, or interpretive PERL, Python, PHP, or may be a shell script. No limits imposed.
  11. Execute ConfigMake to show the contents of an executable script file (vs binary files)
    • ConfigMake -show aSourceFile.sh aVendor aCategory
    • ConfigMake -show /usr/local/bin/aSourceFile.sh
      • Show the contents of an executable script file located in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/vendor/aVendor/aCategory/aSourceFile.sh
      • Else, show the contents of an executable script file located in the current working directory if it contains a ConfigThis directory.
        • ./ConfigThis/vendor/aVendor/aCategory/aSourceFile.sh
      • Else, show the contents of an executable script file located in the directory specified and the aVendor and aCategory parameters are ignored
        • /usr/local/bin/aSourceFile.sh
      • The system presents a prompt to edit the file using the default editor (i.e. vim), respond Y or N
  12. Execute ConfigMake to list project directories
    • ConfigMake -projects
      • ./ConfigThis/aProject/
      • ./ConfigThis/bProject/
      • ./ConfigThis/cProject/
  13. Execute ConfigMake to create project directories (pick one or more of the following)
    • ConfigMake -default
      • ./ConfigThis/aProject/dev_01/config_1000_os/
      • ./ConfigThis/aProject/test_01/config_1000_os/
      • ./ConfigThis/aProject/prod_01/config_1000_os/
        • Creates default directories in ./ConfigThis/aProject/, including three platform _ version directories
        • The login ID owner and default group are retained for all directories
    • ConfigMake -project aProject aPlatform aVersion
    • ConfigMake -project aProject aPlatform aVersion aGroup
    • ConfigMake -project aProject aPlatform aVersion aGroup aOwner
      • ./ConfigThis/aProject/aPlatform_aVersion/config_1000_os/
      • ./ConfigThis/aProject/aPlatform_aVersion/config_2000_tcp/
      • ./ConfigThis/aProject/aPlatform_aVersion/config_3000_usr/
        • aGroup is optional. If aGroup is provided ConfigThis executes chown aOwner:aGroup on all directories
        • aOwner is optional. aOwner defaults to the current login ID (i.e. whoami)
  14. Execute ConfigMake to list all Project links
    • ConfigMake -links aProject | less
    • ConfigMake -links aProject aPlatform aVersion | less
      • List all links for one Project located in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/aProject/aPlatform_aVersion/aCategory1/aLink3.sh
        • /usr/src/ConfigThis/aProject/aPlatform_aVersion/aCategory2/aLink4.sh
      • List all links for one Project in one category located in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/aProject/aPlatform_aVersion/aCategory1/aLink5.sh
        • /usr/src/ConfigThis/aProject/aPlatform_aVersion/aCategory1/aLink6.sh
      • The | less option enables paging of screen output.
  15. Execute ConfigMake to create a link from an executable file to a project
    • ConfigMake -link /usr/local/bin/aSourceFile.sh aCategory aProject aPlatform aVersion
    • ConfigMake -link aSourceFile.sh aVendor aCategory aProject aPlatform aVersion
    • ConfigMake -link -all aVendor aCategory aProject aPlatform aVersion
      • aSourceFile.sh is found in the same manner as described under -file
      • Create a link in the current working directory if it contains a ConfigThis directory.
        • ./ConfigThis/aProject/aPlatform_aVersion/aCategory/aSourceFileLink
      • Else, create a link in the ConfigThis home directory as specified in the ConfigThis.conf file.
        • /usr/src/ConfigThis/aProject/aPlatform_aVersion/aCategory/aSourceFileLink
      • If the link exists then it is listed
      • The -all option creates links to all vendor / category executable files .
      • Links may be created to your other executable files . No need to move or rename files.
      • Files are loaded in alpha order by category directory name and link name (i.e. vs executable file name)

ConfigThis Usage

  1. Prerequisites
    • Presumes prior execution of ConfigMake -this
    • Presumes prior execution of ConfigMake -project...
    • Presumes that directory /home/loginID exists (substitute the appropriate loginID)
    • Presumes that directory /home/loginID/ConfigThis/aProject/aPlatform_aVersion exists
  2. Change current working directory
    • cd /home/loginID
  3. Execute ConfigThis to display the contents of the help file /usr/share/ConfigThis.txt
    • ConfigThis
      • This presumes execution from /usr/local/bin
  4. Execute ConfigThis to display the contents of the conf file /etc/ConfigThis/ConfigThis.conf
    • ConfigThis -conf
  5. Execute ConfigThis to run tests and/or reports for a specified project
    • ConfigThis aProject aPlatform aVersion
    • ConfigThis aProject aPlatform aVersion aCategory
    • ConfigThis aProject aPlatform aVersion aCategory aLink
      • Each link is loaded in turn with output written to
        • /home/loginID/ConfigThis/aProject/aPlatform_aVersion/ConfigThis-aDate-aTime.log
      • If a link returns a non-zero number then a stop file is written to the corresponding category directory
        • /home/loginID/ConfigThis/aProject/aPlatform_aVersion/config_1000_os/config_1000_os_kernal_01.stop
      • Executing ConfigThis without any defined links will cause a non-fatal error per category directory
      • Executing ConfigThis when calling a corrupt or non-working link causes the system to create a stop file in the corresponding category directory
      • Executing ConfigThis with aCategory specified loads only links from aCategory
      • Executing ConfigThis with aLink specified loads only one link
    • ConfigThis aProject aPlatform aVersion -nostop
      • Stop files control the flow of execution within a category directory
      • Executing ConfigThis when a category/*.stop file exists starts execution at the stopped link
      • Executing ConfigThis when a category/last.stop file exists skips the category and continues to the next category
      • Executing ConfigThis with -nostop ingnores stop files and loads all links in all categories
  6. Execute ConfigThis to list all logs
    • ConfigThis -logs aProject aPlatform aVersion
  7. Execute ConfigThis to list all stop files
    • ConfigThis -stops aProject aPlatform aVersion
  8. Execute ConfigThis edit a log
    • ConfigThis -log
    • ConfigThis -log aProject aPlatform aVersion
      • Defaults to the last log created by ConfigThis

Design

  1. This is a sample ConfigThis directory structure with all files:
    • /usr/local/src/webroot/index.html
    • /usr/local/bin/ConfigMake
    • /usr/local/bin/ConfigThis
    • /usr/local/share/ConfigThis/ConfigMake.txt
    • /usr/local/share/ConfigThis/ConfigThis.txt
    • /usr/local/share/ConfigThis/ConfigThis.html
    • /etc/ConfigThis/ConfigThis.conf
    • /usr/local/src/ConfigThis/vendor/bossss/README
    • /usr/local/src/ConfigThis/vendor/bossss/LICENSE
    • /usr/local/src/ConfigThis/vendor/bossss/config_1000_os/conf_1000_os_*.*
    • /usr/local/src/ConfigThis/vendor/currentuserid/config_1000_os/conf_1000_os_*.*
    • /usr/local/src/ConfigThis/MyProject/dev_01/ConfigThis-20000101-235959.log
    • /usr/local/src/ConfigThis/MyProject/dev_01/config_1000_os/link-conf_1000_os_*
    • /usr/local/src/ConfigThis/MyProject/dev_01/config_1000_os/link-conf_1000_os_*.stop
    • /usr/local/src/ConfigThis/MyProject/test_01/config_1000_os/link-conf_1000_os_*
    • /usr/local/src/ConfigThis/MyProject/prod_01/config_1000_os/link-conf_1000_os_*
    • /usr/local/src/ConfigThis/AnotherProject/dev_01/config_1000_os/link-conf_1000_os_*
    • /usr/local/src/ConfigThis/AnotherProject/dev_01/config_1000_os/last.stop
    • /usr/local/src/ConfigThis/AnotherProject/dev_02/config_1000_os/link-conf_1000_os_*
  2. Once executable files are collected and links are created the final structure includes:
    • One or more vendor directories (e.g. aVendor) containing category directories
      • ./ConfigThis/vendor/aVendor/config_1000_os/
    • One executable file (chmod u+x) per test or report program
      • ./ConfigThis/vendor/aVendor/config_1000_os/config_1000_os_01_kernal.sh
    • One or more project directories (e.g. MyProject) corresponding to project, utility, and/or web application source code
      • ./ConfigThis/aProject/
    • One project platform and version directory per target platform (e.g. dev_01)
      • ./ConfigThis/aProject/aPlatform_aVersion/
    • One project platform version config_N000_ directory per category.
      • ./ConfigThis/aProject/aPlatform_aVersion/config_1000_os/
    • One link per vendor executable file
      • ./ConfigThis/aProject/aPlatform_aVersion/config_1000_os/link-config_1000_os_01_kernal.sh
  3. Naming Conventions:
    • All platform_version directories contain an underscore character _ between platform and version
    • There is no significance to the category directory names and link names except as it pertains to the order of execution
    • Category directories are preset to 1000 os, 2000 ip, 3000 usr, 4000 vm, 5000 dev, 6000 db, 7000 app, 8000 mon (monitoring and IDS), 9000 bak (backup)
      • ./ConfigThis/MyProject/dev_01/config_1000_os/
    • Link names start with link- then the category directory name, followed by user specified text
      • ./ConfigThis/MyProject/dev_01/config_1000_os/link-config_1000_os_1233_kernal.sh

Management

  • ConfigThis platform _ versions pertain to configurations and do not necessarily correspond to applications, utilities, and/or project versioning (e.g. MyProject v1.32.14, ConfigThis version dev_03, test_02, prod_02)
  • ConfigThis order of execution is determined by each category directory name and link name within each project / platform _ version directory, regardless of vendor executable file name.
    • A group of low level category directories (e.g. config_1000_os, config_2000_tcp, etc.) may be copied to each new project, where most additions and changes will be made to other higher level category directories (e.g. config_7000_app ). Thus all / project / platform _ version directories will contain the same set of category directories
    • As an alternative, you may group low level categories into a separate / project / platform _ version directory targeting a commonly used platform with, for example, a database server and multiple web applications:
      • ./ConfigThis/AllProjects/dev_low/...
      then group high level categories into separate / project / platform _ version directories targeting each database server and/or web application:
      • ./ConfigThis/MyProject/dev_high/...
      • ./ConfigThis/AnotherProject/dev_high/...
    • Corresponding logs (i.e. test results) are separately maintained per project / platform_version directory.
    • Once low level tests have been passed you may repeat testing exclusively with the higher level tests.
  • ConfigThis no harm done policy:
    • Executable files should do no harm - applicable or not
    • Empty project / platform _ version category directories are permitted.
    • Executing non-performative links is permitted (e.g. testing for a database when no database is required). However, a stop file may be generated for that category and other categories are subsequently loaded.
    • Non-executable files may be stored in ConfigThis directories (e.g. README), each will be ignored by ConfigThis during processing.
  • It is highly recommended that you not store the ConfigThis directories and files in your webroot directory or similar source code directory. Otherwise, it is up to you to decide how to manage links and executable files .
  • rsync may be used to copy project / platform _ version directories of links or / vendor executable files locally and remotely
    • cd /home/loginID
    • rsync -av --filter='config*' --exclude=config_7* ./ConfigThis /tmp/ConfigThis
